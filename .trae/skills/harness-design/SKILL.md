# Agent Harness 设计哲学(通用护栏)

> 一个 harness 的意义,是把一个通用模型变成一个能干某件事的 agent。而它干这件事的本事,大半不在模型权重里 —— 在你给它的**工具、上下文、反馈、文档**里,也就是 harness 里。
>
> 所以每往里加一样东西,先问一句:它是在**放大模型自己的推理**,还是在**替模型把活干了**?前者让 agent 够得着你没预见的情况;后者把 agent 的天花板焊死在你此刻的想象力上,并让整个 harness 一遇到没预设的世界就脆。
>
> 你现在正运行在一个 harness 里(这段 system prompt 就是),所以这些不是抽象 —— 是你天天被设计、也在设计别人的东西。

真正干活的 harness 常常**应该**替模型解决某些事(确定性的、稳定的、安全相关的),所以这里的边界更细 —— 不是「永不替模型干活」,而是「在对的层、用对的代价干」。

## When to Use This Skill

动手**之前**读它,只要你要碰下面任何一样:

- 写 / 改 **system prompt、agent 指令、角色设定**。
- 定义或描述 **tool**(工具的能力边界、入参、description / docstring)。
- 写 **agent 要读的文档 / 指南 / 手册**。
- 设计**注入 loop 的上下文**:每轮喂给模型的状态、记忆、检索结果、环境快照。
- 设计 **反馈 / 可观测性**:报错文案、tool 返回、trace、日志、eval、赛后诊断。
- 编排 **子 agent / 多 agent**(谁看到什么、谁替谁做决定)。
- **review** 上述任何 diff,或判断「某个修复 / 特性到底值不值得做、对谁有用」。

## 原则一:放大推理,而非替代推理(镜子 vs 拐杖)

每一个「给 agent 的东西」,先归类 —— 它喂进 loop 替模型**做了决定**,还是给模型/给你**看清了状态**?

| | 镜子 Mirror | 拐杖 Crutch |
|---|---|---|
| 出口 | **事后 / loop 外**(下一步、赛后、给设计者) | **对局内 / loop 中**(当下这一步喂给模型) |
| 作用 | 让模型(和你)看清发生了什么、为什么 | **替**模型解决一个本该它推理的判断 |
| 对能力 | **放大**——agent 能处理你没预见的情况 | **封顶**——agent 永远不会比你写死的答案更好 |
| 对健壮 | 没有上限 | 世界一偏离你的假设就**脆** |

**核心一句**:拐杖把 agent 的上限焊在**你此刻的预见**上。镜子没有这个上限 —— 它让 agent 在你从没想过的局面里也能变强。这就是你为什么用模型而不用脚本:别把脚本式的写死答案,塞回模型的 loop。

### 关键技巧:同一份数据,挑对出口

底层数据常常能走两个出口 —— **优先走镜子那个**。

- 一次 tool 失败 → ① 进 trace / 赛后诊断(**镜子**,留);② 把「失败原因 + 你下一步该怎么做」拼进当下报错喂回模型(**半个拐杖**)。
- 设计时先问:「能不能只走 loop 外那个出口?」能,就别开 loop 内的。

### 「抹平摩擦」不是免费的

每抹平一个困难,你也**同时停止了测量它**。静默重试、静默纠错、自动兜底 —— 都让 agent 当下顺了,但你和模型都对「它本来会在这儿栽」失明,模型也永远练不出处理它的本事。

- 抹平之前先问:这摩擦是**该消灭的体力活**(确定性、稳定、模型只会徒增错误 —— 解析、格式化、幂等重试),还是**该暴露的判断点**?
- 真要抹平,至少**把摩擦导向镜子**:把重试次数、纠错率打进 trace / 指标。否则一个本该报警的退化信号被你悄悄吃掉了。

### 取舍:替模型干活,什么时候是对的(这条想清楚,别凭手感)

替模型干活(在 harness 里把问题解掉)**有合法场景**,别教条地一律不做:

- **该在 harness 里解**:问题定义清晰、稳定、确定性强,模型介入只会加错和加成本 —— 解析、校验、重试、分页、幂等、纯格式化。这是**基建**,不是拐杖。
- **该留给模型**:需要判断、需要随情境适配、你列不全所有 case 的 —— 这正是模型存在的理由。把这种判断写死在 harness 里 = 拐杖。
- 判别尺:**你是在替它做体力活,还是在替它做判断?** 体力活尽管接管;判断,交还给它。

## 原则二:信息高度 —— 传事实与模型,慎传祈使

写进 agent 上下文(prompt / tool 描述 / 文档)的每一句话,都落在三个高度之一:

| 层 | 是什么 | 例子 | 裁决 |
|---|---|---|---|
| **L1 事实 / 契约** | 存在什么、什么意思、tool schema、不变式、地面真相 | 「`cone` 是 ±60°」「本项目用 pnpm」「这个 API 用 Bearer 鉴权」「这个函数会改全局状态」 | **必传** —— 没它模型无法推理 |
| **L2 系统模型 / 涌现含义** | 机制的非显然后果、心智模型、各部件如何互相作用、**作为系统属性的**失败模式 | 「重试不幂等会双写」「检索结果带时间戳、越旧越可能过期」「这两个工具的副作用会互相覆盖」 | **该传**,用**中性「系统会怎样」措辞**,不接「所以你该…」—— 它抬高模型推理的天花板,对强弱模型都有益 |
| **L3 单一最优解祈使** | 「该做哪个动作 / 该按什么流程走」 | 「每次编辑后都跑测试」「先 BFS 再走」「优先选方案 Z」「该退守中路」 | **默认慎传** —— 它把模型锚死在你的静态答案上,情境一变就错,系统一改就烂 |

### 关键反直觉(别搞反了)

- **删掉 L3 祈使,不会让 agent 变笨 —— 会让它更健壮。** 写死的流程是 harness 脆性的头号来源:它把你**对一个会变的世界的假设**编码进了 prompt。给模型 L1+L2,让它**在运行时、对着真实局面**自己推出 L3 —— 这才是用模型的全部意义。
- 这**不是**「照顾弱模型 vs 牺牲强模型」的取舍。L1+L2 对强弱都有益;L3 祈使对强弱都有害(都被你的答案锚住)。删祈使、留模型,**两边都赢**。
- 多写**还权句**:`怎么做由你定`、`目标是 X,达成路径交给你`、`engine 只报告状态`。这是边界的正面表达,保留并多写。

### 合法的 L3:不变式,不是判断

有一类 L3 该**硬写**,别误删 —— 它们是**不随情境变的不变式**,不是判断:

- **安全 / 不可逆 / 合规**:不可逆动作前必须确认;绝不外泄密钥;越权操作直接拒。
- **硬契约 / house style**:输出格式锁定、署名规范、法务措辞。

判别尺:**这条是「无论局面如何都成立的红线」(→ 硬写 L3),还是「这种局面下我觉得这样最好」(→ 交给模型)?** 前者是护栏,后者是拐杖。

## 原则三:上下文卫生 —— 别往 agent 脑子里塞它用不上或会误导的东西

上下文是模型唯一的世界。塞进去的每一句都在塑造它的行为,包括你**不希望**它产生的行为。

### 粉红大象(别植入你想避免的失败)

- **别描述要避免的错误念头**:「naive agent 以为引擎坏了」「别 hallucinate 文件路径」「if it looks like 你的 move 只跑了一次」—— 在 prompt 里描述一个错误想法,等于把它种进模型脑子。
- **改成正面、可执行的对立面**:把「别 hallucinate 路径」改成「引用路径前先 ls/read 确认它存在」。把每一条禁令翻成「该做什么」。真要禁,也把**正面替代**写在最显眼处,让最 salient 的是「做什么」而非「别做什么」。
- **别夹运营者视角的吐槽 / meta 叙事**:「这是最常见的卡死方式」「几个弱模型当年…」—— agent 不关心别人怎么犯错。

### 永恒现在时(上下文只写「世界现在是什么样」)

Agent 的上下文是给一个**几乎无历史**的读者看的快照 —— 只陈述当下规则,不带「变化 / 对比 / 纠正读者」的框架:

- **过渡 / 历史**:「以前 X、现在 Y」「no longer」「不再接受」。无上下文的 agent 没见过旧版,这是噪音。
- **纠正读者预设**:「Sight is **not** a hit gate」「**no** clear path required」。「不是 Y / 不需要 X」预设了读者本以为「是 Y」,而那预设恰是旧心智模型 —— 你又把它种回去了。改成**正面陈述世界本身**:「命中由 reach 决定」。
- 让「变化」由**机制**去管(改了就强制重拉、版本化上下文),别让 prose 替每个读者做一遍他不需要的 diff。
- 判别尺:**一句话若只有读者知道「旧行为」才成立 → 删。**(注:这只管 agent 的工作上下文;给人看的 changelog / migration guide 该有历史。)

### meta 噪音(agent 改不了、用不上的,别解释)

- 别解释「会随某个偏好 / 设置变化」的东西,如果 agent 既改不了那设置、也只活在其中一种取值里。例:「display 字段按 owner 语言渲染、默认中文」—— agent 用不上。
- 判别尺:**告诉它该*产出*什么(`reply in 中文`)= 留;解释它*收到*的会随某设置*变化* = 删。** 只留可执行的对立面:「机器键稳定、display 不稳定 → 绑机器键」。

## 原则四:判断的地基 —— 以真实行为为准 + 诚实契约

- **只认真实行为(代码 / trace / transcript / eval),不认 agent 自述 + 你的假设。** Agent 会对自己的行为产生幻觉(「我跑了 X」其实没跑)。判断「某特性是否必要 / 有用」,看**真实的高分轨迹/日志怎么跑**,不是看叙事、也不是看你脑补的用户。
- **大部分「防呆 / 健壮性」特性是给最弱 case 的扶手 —— 对强 case 冗余甚至有害。** 在一个成熟的主路径上加一段「几乎不触发」的新逻辑,风险常 > 收益。要加,**明确定位成 fallback / 训练轮**,别让它悄悄成为所有人的默认。
- **镜子(可观测性 / 诊断)是少数对所有水位都有用的东西** —— 它让所有人(含最强的)看清自己。但前提是它**不撒谎**:一个误报的指标比没有指标更坏。
- **契约必须诚实**:harness 暴露的工具 / 字段,必须真能用。一个文档里承诺、实际坏掉(永远返回 undefined / flag 不存在)的契约,**比没有更坏** —— 模型会**当它能用**去推理,然后静默失败。
- **撤掉一个能力,必须同步撤掉所有对它的引用**(prompt / 文档 / tool 描述里),`grep` 一遍。否则你又在承诺一个不存在的东西。

## 动手前 Checklist(逐条过)

1. **放大还是替代?** 这东西喂进 loop 替模型做决定(拐杖),还是给它/给你事后看清(镜子)?能不能只走 loop 外的出口?
2. **它替模型解决了一个本该它推理的判断吗?** 是 → 它把 agent 上限封在你的预见上,默认停。(替**体力活**可以,替**判断**不行。)
3. **真实 trace / eval 里,这问题对强 case 真存在吗?** 还是只在弱 case?只扶弱 → 明确标成 fallback,别成默认。
4. **(写上下文)这句是 L1 事实 / L2 系统模型(留)还是 L3 祈使(慎)?** L3 只为「不随情境变的红线」(安全 / 不可逆 / 合规 / 硬格式)保留;判断类的删,交给模型。
5. **有没有粉红大象 / 非永恒现在时 / meta 噪音?** 植入要避免的失败、运营者吐槽、「以前 X 现在 Y」、「不是 Z」、解释 agent 改不了也用不上的设置 —— **只有读者知道旧行为才成立的句子** → 删,改正面陈述当下世界。
6. **契约诚实吗?** 暴露的工具 / 字段都真能用?撤掉的能力引用都清干净了?(`grep`)
7. **这判断基于真实行为,还是 agent 自述 / 你的假设?** 后者 → 回去看代码 / trace / transcript。

## 判例库(patterns —— 后续照这个对齐)

| 改动 / 模式 | 归类 | 裁决 |
|---|---|---|
| 工具报错只回事实(`move failed` / `file not found: /x`) | 镜子(事实) | **留** |
| 工具失败 → 进 trace / 赛后诊断 | 镜子 | **留**,对所有水位有用 |
| 工具报错附「下一步该跑什么」(`did you mean /y? run Z next`) | 半个拐杖(in-loop 祈使) | 候选事实留;「该跑 Z」那半句**删** —— 让模型自己定 |
| 把「你已试过 X 且失败」作为**状态**注回上下文 | 镜子 / 状态(不是祈使) | **留** —— 是事实状态,不是替它做决定 |
| 把算好的「最优计划」注入,让 agent 照着执行 | 拐杖(封顶在你的预见) | **默认不做**;给事实 + 目标,让它自己规划。除非是真正固定的流程 |
| 静默重试 flaky 工具 3 次 | 抹平摩擦 | 可做,但把重试**记进 trace**(摩擦导向镜子),否则你对 flakiness 失明 |
| 静默纠正模型产出的坏 JSON | 抹平摩擦 | 慎:至少打一个指标,否则掩盖了模型在退化的信号 |
| system prompt:「always run tests after every edit」 | L3 祈使 | **重审**:除非是硬闸门,改成 L2 事实(「CI 门禁是 `cargo check`;未验证的改动可能挂构建」),让模型自行决定 |
| system prompt:「本项目用 pnpm;`package-lock.json` 是脏的、忽略它」 | L1 事实 | **留** |
| 文档:「该 API **不再**接受旧 auth header」 | 非永恒现在时 | **改正面**:「该 API 用 `Authorization: Bearer` 鉴权」 |
| 文档:「**don't** hallucinate 文件路径」 | 粉红大象 | **改正面**:「引用路径前先 ls/read 确认存在」 |
| tool 描述承诺一个工具其实没有的 `--json` flag | 不诚实契约 | **修工具或修描述**;绝不发幽灵契约 |
| 每条 tool 结果都附「Tip: 用 X 更快」 | 对局内噪音 / 祈使 | **移进读一次的文档(L2)**,别每轮在 in-loop 重复 |
| 不可逆动作前必须确认;绝不外泄密钥 | **合法 L3**(不变式) | **硬写** —— 这正是 L3 该用的地方:安全 / 不可逆 / 合规,不随情境变 |

## 一句话记住

**给 agent 一面诚实的镜子,加一套真实的事实和系统模型,让它自己推理出怎么做;别把你此刻的答案焊进它的 loop —— 那只会让 harness 脆、让 agent 停在你想象力的边界上。体力活尽管替它做,判断,交还给它。**
